# 10 Dockerfile Principles for Scientific Computing (HPC)

These principles are embedded in all templates generated by `generate_dockerfile.py`.
When generating a Dockerfile outside of the templates (LLM fallback), apply all 10.

---

## 1. Pin the base image by digest
Use `FROM ubuntu:22.04@sha256:<digest>` not `:latest`.
Silent upstream changes break reproducibility without any build error — you get
a different environment than last month with the same tag. Pin the digest.

> For well-maintained scientific base images (nanoporetech/dorado, bioconductor_docker)
> pinning the tag is acceptable; still prefer digest for production images.

## 2. mamba/conda over apt for scientific packages
apt lacks version resolution for bioinformatics; it can't handle CUDA linking,
R/Python ABI interop, or complex dependency trees that tools like samtools or
PyTorch require. Use mamba for all scientific packages. Use apt only for
system-level deps (curl, procps, libssl-dev) that conda doesn't provide.

## 3. Clean caches in the same RUN layer
```dockerfile
# CORRECT — cache cleared in the same layer
RUN mamba install -y samtools=1.21 && mamba clean --all -y

# WRONG — separate RUN doesn't reduce image size (layers are additive)
RUN mamba install -y samtools=1.21
RUN mamba clean --all -y
```

## 4. OCI labels on every image
Every image must have at minimum:
```dockerfile
LABEL org.opencontainers.image.title="samtools" \
      org.opencontainers.image.version="1.21" \
      org.opencontainers.image.maintainer="sahuno" \
      org.opencontainers.image.description="samtools 1.21 for HPC"
```
This makes images self-documenting in `apptainer inspect` and Docker Hub.

## 5. Validation layer as the final build step
```dockerfile
RUN samtools --version
```
If the install is broken, the build fails loudly here rather than producing
a silently broken image that fails at runtime on the HPC node.

## 6. Non-root user for HPC compatibility
```dockerfile
RUN useradd -m -s /bin/bash biouser
USER biouser
```
HPC bind mounts and shared filesystems have permission edge cases with root
containers. Running as a non-root user avoids file ownership problems when
writing outputs to `/data1/` or similar NFS mounts.

## 7. Explicit ENV for PATH and library resolution
```dockerfile
ENV PATH=/opt/conda/bin:$PATH
ENV LD_LIBRARY_PATH=/opt/conda/lib:$LD_LIBRARY_PATH
```
Shell profiles (`~/.bashrc`, `/etc/profile`) are not sourced in non-interactive
container execution (e.g., `apptainer exec container.sif samtools view ...`).
PATH must be set via ENV so it applies in all execution modes.

## 8. Cache-friendly layer ordering
Order layers from least-likely-to-change to most-likely-to-change:
1. System apt packages (rarely change)
2. conda/mamba environment (changes on version bumps)
3. pip packages (changes more often)
4. Config files, scripts (change most often)

This maximises GitHub Actions layer cache reuse — only changed layers rebuild.

## 9. Multi-stage builds for compiled tools or proprietary binaries
For tools that require compilation (C++, Rust, Go) or tools distributed as
pre-built tarballs that cannot be installed via conda/pip (e.g. CellRanger,
STARsolo, Guppy, GATK bundles):

**Compiled tools:**
```dockerfile
FROM ubuntu:22.04 AS builder
RUN apt-get install -y build-essential cmake ...
COPY . /src
RUN cmake /src && make -j4

FROM ubuntu:22.04
COPY --from=builder /usr/local/bin/mytool /usr/local/bin/mytool
```

**Proprietary pre-built binary (e.g. CellRanger):**
```dockerfile
FROM ubuntu:22.04
# Install system deps
RUN apt-get update && apt-get install -y --no-install-recommends \
    curl ca-certificates \
    && apt-get clean && rm -rf /var/lib/apt/lists/*
# Download and extract binary
ARG VERSION=8.0.0
RUN curl -sL "https://example.com/cellranger-${VERSION}.tar.gz" \
    | tar -xz -C /opt \
    && ln -s /opt/cellranger-${VERSION}/cellranger /usr/local/bin/cellranger
ENV PATH=/opt/cellranger-${VERSION}:$PATH
# Validation
RUN cellranger --version
```
Key points for proprietary binaries:
- Use `ARG VERSION` so the same Dockerfile rebuilds for new releases by passing `--build-arg`
- Download in one RUN layer and clean in the same layer
- Add binary directory to `ENV PATH` explicitly (principle 7)
- If the download URL requires authentication (10x Genomics), note this in a comment
  and instruct user to use a `--build-arg DOWNLOAD_URL=<signed-url>` pattern
- Always include the validation `RUN` immediately after install (principle 5)

## 10. ENTRYPOINT + CMD split
```dockerfile
ENTRYPOINT ["samtools"]
CMD ["--help"]
```
- `ENTRYPOINT` sets the tool as the container's executable
- `CMD` provides the default argument (shown when run with no args)
- Together they work correctly in Snakemake `container:` directives:
  `samtools view -b input.bam` → `docker run sahuno/samtools view -b input.bam`
- Don't use `ENTRYPOINT` as a shell script unless the tool genuinely needs it
