- **`DSS::DMLtest` silently corrupts results under SLURM memory pressure**: `DMLtest` defaults `ncores = max(detectCores() - 3, 1)` and `detectCores()` returns physical cores on the node (64 on componc_cpu), **ignoring the SLURM cgroup cpu allocation**. `DSS:::dispersion.shrinkage.BSseq` then calls `mclapply(..., mc.cores=ncores)`. When fork workers exceed the cgroup memory limit the kernel OOM-kills them; `mclapply` returns NULLs for their tasks; `unlist(shrk.phi2)` silently drops NULLs; the subsequent assignment `shrk.phi[ix] <- shrk.phi2` **recycles** the short phi vector R-style, mis-assigning dispersion estimates to wrong CpGs. The script continues to `=== DONE ===` and writes outputs — easy to miss. DMR counts can inflate 10–20× over the true value. Confirmed 2026-04-24 on mm10 ONT data: a 40-DMR "result" on chr19 collapsed to 4 DMRs once ncores was bounded. **Fix**: always pass explicit `ncores = as.integer(Sys.getenv("SLURM_CPUS_PER_TASK", "1"))` to `DMLtest(..., ncores = ncores)`. Never rely on DSS's default.
- **Detection**: any DSS run where any of the following are true must be re-run, not trusted:
    - `sacct -j <id>` shows `OUT_OF_MEMORY`
    - stderr shows `oom_kill events: N` for N > 0
    - stdout/log contains `number of items to replace is not a multiple of replacement length` (appears near the end of DMLtest; shows up twice — once for phi1, once for phi2)
- **`BiocParallel::register(SerialParam())` does NOT help**. DSS's dispersion shrinkage uses `mclapply` from `parallel`, not `bplapply` from BiocParallel. Only the BSmooth smoothing step respects the BiocParallel backend. To serialize DSS you must pass `ncores=1` to `DMLtest`.
- **DMLtest results are genome-size dependent**: `est.prior.BSseq.logN` estimates the empirical-Bayes dispersion prior from all CpGs in the BSseq object; `compute.mean.Smooth` uses the global mean methylation as a smoothing anchor. A run on one chromosome is therefore not expected to reproduce the same-chr slice of a whole-genome run even when the input data is bit-identical. This is **not** a bug — but it means: (a) chr-by-chr sharding requires merging before `callDMR`, and (b) you cannot validate a genome-wide run against a chromosome-only run at the DMR level, only at the DML-test row-count level.
- **Post-run verification (mandatory)**:
    1. `sacct -j <id> --format=State` must be `COMPLETED` — treat `OUT_OF_MEMORY` as a failure even if outputs exist.
    2. `grep -c "number of items to replace" <stdout>` must be 0.
    3. DML-test per-chromosome row counts must match the max per-sample input count for that chromosome (±0.01% on autosomes; chrX/chrY tolerate larger deltas from sex-chromosome coverage asymmetry).
- **`withCallingHandlers(DMLtest(...), warning=...)` is required** to route DSS's stderr warnings into the log file. `globalCallingHandlers(message=...)` alone catches `message()` only, so the recycling warning slips past into the SLURM stderr and is easily missed.
- **Memory planning for DSS on ONT data**: with `ncores=N` and whole-genome mm10 input (~21M CpGs × 6 samples), peak memory is roughly `base_rss + N × copy_on_write_delta`. A working recipe on MSKCC HPC: `--cpus-per-task=8 --mem=128G` with `ncores=8`. Do not scale cpus without scaling memory — each fork is a near-full copy of the BSseq object under COW if the child writes back to shared pages.
