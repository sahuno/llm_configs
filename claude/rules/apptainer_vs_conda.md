- **For short, high-fan-out HTSlib/samtools/bcftools/htslib-tool invocations on MSKCC HPC, prefer the apptainer SIF over the conda binary.** Cause: `/home/ahunos/miniforge3/envs/...` sits on NFS. On a fresh compute node (the default in Snakemake/Nextflow per-sample rules) the dynamic loader has to do an NFS attr-lookup + read for *every* shared library on first touch (libhts, libdeflate, libcurl, libcrypto, libz, etc.). The SIF is one Weka-resident image — one fast read, then everything is in RAM.
- **Empirical magnitude (samtools sort, 5 reps each, identical -@32 -m2G -l1, fresh nodes, 2026-04-28)**: conda mean wall 16.4 s, CV 13.3 %, **median 1.06 M minor page faults** (some runs 2.2 M); container mean wall 14.0 s, CV 3.2 %, median 43 k minor page faults. Conda's wall scales linearly with minor PF count (~2.5 µs per fault) — a clean cold-start tax, *not* container vs native overhead. Welch t = -2.35.
- **How to apply in pipelines**: in Snakemake/Nextflow rules wrapping samtools/bcftools/htslib/modkit/dorado, use the `container:` / `singularity:` directive (pointing at the SIF in `/data1/greenbab/users/ahunos/apps/containers/`) instead of `conda:`. For long-running interactive shells where the binary's libraries are warm in cache, conda is fine — the cold-start tax is one-time per node-process.
- **This is a cluster-storage-topology effect, not a Linux container effect.** On clusters where the conda env lives on a fast parallel FS, the gap may disappear. Don't generalise the recommendation off-cluster.
- **Container catalogue**: `/data1/greenbab/users/ahunos/apps/containers/`. Authoritative paths registered in the user's `softwares_containers_config.yaml` (referenced from CLAUDE.md §6 Containers). Pass that wrapper's path as `$SAMTOOLS_BIN` and downstream code is none the wiser.
